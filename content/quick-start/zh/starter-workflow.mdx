第一个工作流的目标，不是震撼别人，而是证明 OpenClaw 能稳定产出一个你能快速检查的结果。它应该像一个试车场，而不是终局系统。

渠道已经通了，说明消息能进来，也能出去。接下来要验证的是：当消息进来之后，**流程能不能按预期跑完**。

## 推荐的第一个工作流：链接摘要

链接摘要是入门首选，原因有四：

- 触发方式简单：用户发一条 URL
- 依赖最少：不需要额外的外部账号或认证
- 结果可验证：你看一眼摘要就知道对不对
- 出错容易定位：要么 URL 没抓到，要么模型没调用，非常清晰

### 完整配置步骤

#### 第一步：在 AGENTS.md 里添加行为规则

打开 `workspace/AGENTS.md`，加入以下内容：

```markdown
## 链接摘要规则

当用户发送一个 URL 时：

1. 抓取该页面的主要内容
2. 识别页面语言
3. 用中文生成一段摘要（150-300 字）
4. 回复格式如下：

**摘要**
[摘要内容]

**来源**：[原始 URL]
**语言**：[页面原始语言]
```

#### 第二步：在配置中启用 web-summary 技能

打开 `openclaw.config.json`，在 `skills` 数组中加入：

```json
{
  "version": "1",
  "agent": {
    "provider": "openai",
    "model": "gpt-4o-mini",
    "systemPrompt": "workspace/AGENTS.md"
  },
  "workspace": {
    "root": "./workspace"
  },
  "channels": [
    {
      "type": "telegram",
      "token": "${TELEGRAM_BOT_TOKEN}",
      "mode": "webhook",
      "webhookPath": "/webhook/telegram"
    }
  ],
  "skills": [
    {
      "name": "web-summary",
      "enabled": true
    }
  ]
}
```

#### 第三步：重启 OpenClaw

```bash
openclaw start
```

启动日志现在应该显示技能已加载：

```
[openclaw] Starting OpenClaw v0.x.x
[openclaw] Loading config: ./openclaw.config.json
[openclaw] Provider: openai (gpt-4o-mini)
[openclaw] Authenticating with OpenAI... OK
[openclaw] Channel: telegram — webhook registered at /webhook/telegram
[openclaw] Skills: web-summary (enabled)
[openclaw] Ready.
```

#### 第四步：发送测试链接

去 Telegram 找到你的机器人，发送一个链接：

```
https://openai.com/blog/gpt-4o
```

预期回复格式：

```
**摘要**
GPT-4o 是 OpenAI 推出的最新旗舰模型，能够同时处理文本、音频和图像输入……

**来源**：https://openai.com/blog/gpt-4o
**语言**：英语
```

如果看到这样的格式化回复，第一个入门工作流已经跑通。

## 你真正在验证什么

这个工作流虽然简单，但它同时验证了整条链路的每一个环节：

```
[用户发送 URL]
      │
      ▼
[Telegram 推送事件到 Webhook]
      │
      ▼
[OpenClaw 接收消息]
      │
      ▼
[消息路由器识别触发类型]
      │ 检测到 URL → 调用 web-summary 技能
      ▼
[web-summary 技能抓取页面]
      │
      ▼
[代理调用模型生成摘要]
      │
      ▼
[结果按 AGENTS.md 模板格式化]
      │
      ▼
[回发到 Telegram 群聊]
```

任何一环失败，你都能在日志里快速定位。

## AGENTS.md 写法详解

`AGENTS.md` 是 OpenClaw 代理行为的核心文件。它告诉代理：

- 应该做什么
- 不应该做什么
- 碰到某类输入该怎么处理
- 输出格式长什么样

一个良好的入门 `AGENTS.md` 结构：

```markdown
# 角色定义

你是一个信息整理助手，负责帮用户处理链接、文章摘要和简报。

# 行为规则

## 链接处理
- 收到 URL 时，抓取页面主要内容
- 用中文生成摘要（150-300 字）
- 按照指定格式回复

## 通用规则
- 如果不确定用户意图，先询问确认
- 摘要只提炼关键信息，不添加推测
- 每次执行完成后，向用户确认结果

# 权限范围
- 可以读取公开网页
- 可以写入 workspace/logs/ 目录
- 不可以访问需要登录的内容
```

> 不要一次写太多规则。第一版只需要覆盖你要验证的那一个场景。

## 选择第一个工作流的标准

如果你不打算做链接摘要，也可以选其他方向。判断标准只有一个：

**这个任务能不能在 5 分钟内完成一次完整测试？**

符合这个标准的，通常是：

| 任务类型 | 触发方式 | 输出 | 验证难度 |
|---------|---------|------|---------|
| 链接摘要 | 用户发 URL | 格式化摘要 | 低 |
| 文本转要点 | 用户粘贴文字 | 结构化要点 | 低 |
| 每日简报 | 定时（cron） | 简报发到群聊 | 中 |
| 简单问答 | 用户提问 | 直接回答 | 低 |
| 格式转换 | 用户发内容 | 转换格式回复 | 低 |

不符合标准的，通常是：

- 需要 3 个以上外部 API
- 依赖长时间积累的上下文或记忆
- 触发条件不明确或不稳定
- 输出格式复杂，人工验证成本高

## 第一条工作流的验证清单

跑通之前，确认以下几点：

- 配置文件里的技能名称和 OpenClaw 内置名称完全一致
- `AGENTS.md` 对这类输入有明确的处理规则
- 触发渠道已经正常连接（消息能双向传递）
- 日志里能看到技能调用记录

跑通之后，确认以下几点：

- 结果格式符合 `AGENTS.md` 里的模板
- 日志里有完整的执行记录（从触发到回复）
- 你能在 3 步以内重现同样的结果

## 如果工作流没有按预期执行

### 消息发进去，机器人没反应

先排除渠道问题：

```bash
openclaw status
# 确认进程在运行

curl "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getWebhookInfo"
# 确认 Webhook 已注册且 URL 可达
```

### 机器人回了，但没有调用技能

检查日志：

```bash
openclaw start --log-level debug
```

在 debug 日志里搜索 `skill`，确认技能调用是否出现：

```
[openclaw] Skill invoked: web-summary
[openclaw] Fetching URL: https://openai.com/blog/gpt-4o
```

如果没有这行日志，说明代理没有识别到该输入应该调用技能。检查 `AGENTS.md` 里的规则是否足够明确。

### 技能调用了，但返回结果为空或报错

```
[openclaw] Skill error: web-summary — unable to fetch page content
```

可能原因：
- 页面需要登录（web-summary 只能抓取公开页面）
- 页面有反爬虫保护
- 网络连接问题

换一个更简单的公开页面重试：

```
https://example.com
```

### 结果回来了，但格式不对

检查 `AGENTS.md` 里的格式模板，确认：

1. 格式描述清晰（代理需要明确的指令，不能靠"应该"来猜）
2. 没有歧义（比如"简短"到底是 50 字还是 300 字，最好直接写数字）

## 先别急着做这些复杂场景

下面这些方向很强，但不适合做第一条工作流：

- 多 Agent 协作系统
- 跨多个 SaaS 的长链路自动化
- 需要大量记忆与长期状态管理的流程
- 一次执行几十步的内容工厂
- 需要浏览器登录态、复杂权限和多渠道分发的系统

这些场景不是不能做，而是应该放在"你已经有一条跑通的起步流程"之后。

## 跑通后应该记录什么

当第一条工作流成功时，顺手记下这三样东西：

- **触发方式**：发什么触发它（URL / 关键词 / 定时 / 命令）
- **成功输出样例**：截图或复制一条真实的成功回复
- **第一排错点**：出了问题你第一个看哪里（日志 / 渠道 / 技能调用记录）

这会成为你后面扩展模板、技能和定时任务时的基准样本。更复杂的流程出了问题，可以回退到这条"最短可用路径"对比。

## 更好的下一步

当一条工作流稳定跑通后，你可以进入 **Learn** 补强认知，或者去 **Templates** 选择更贴近目标的场景模板。如果它还没跑通，先把范围压缩得更小，不要急着往外扩。
