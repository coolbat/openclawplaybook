很多人第一次接触 OpenClaw，会先把它理解成"能接入飞书或 Telegram 的大模型聊天机器人"。这个理解不算全错，但会把你带偏。

OpenClaw 真正有价值的地方，不是多一个聊天窗口，而是让代理**通过渠道接收输入、调用工具或技能、执行可重复流程，并把结果送回固定位置**。如果你只把它当聊天工具，你会一直在调 prompt；如果你把它当工作流执行系统，你才会开始关心渠道、权限、上下文、输出位置和复用性。

## 最短的定义

把 OpenClaw 理解成一个**带 AI 接口的工作流执行层**，而不只是 LLM 包装器。

- 聊天消息通常只是触发入口，不是全部产品。
- 模型负责理解与推理，但技能、命令、集成和工作区决定它到底能做什么。
- 真正的目标不是"回了一句像样的话"，而是"这件事明天还能再跑一次"。

> 如果任务本身不可重复、没有渠道入口、也不产出明确结果，那它可能不是 OpenClaw 最好的第一场景。

## 系统结构：它由哪些部分组成

理解 OpenClaw 最快的方式，是看它的核心组件是怎么连在一起的：

```
┌─────────────────────────────────────────────┐
│                  渠道层 (Channels)            │
│   飞书 / Telegram / CLI / Webhook / 定时任务  │
└──────────────────────┬──────────────────────┘
                       │ 触发 / 输入
                       ▼
┌─────────────────────────────────────────────┐
│                OpenClaw 核心                  │
│  ┌──────────────┐    ┌─────────────────────┐ │
│  │  消息路由器   │───▶│   代理执行引擎        │ │
│  └──────────────┘    └──────────┬──────────┘ │
│                                 │             │
│  ┌──────────────────────────────▼──────────┐ │
│  │              工具 / 技能层               │ │
│  │  搜索 / 摘要 / 写作 / 代码 / 外部 API   │ │
│  └──────────────────────────────┬──────────┘ │
│                                 │             │
│  ┌──────────────────────────────▼──────────┐ │
│  │              工作区 (Workspace)          │ │
│  │  AGENTS.md / 记忆 / 文档 / 任务状态     │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────┬──────────────────────┘
                       │ 输出
                       ▼
┌─────────────────────────────────────────────┐
│                  输出目标                     │
│   群聊回复 / 飞书文档 / 通知 / 结构化数据      │
└─────────────────────────────────────────────┘
```

每个部分都有自己的职责：

| 组件 | 职责 | 典型配置 |
|------|------|----------|
| **渠道 (Channel)** | 消息的入口和出口 | 飞书机器人、Telegram Bot、CLI |
| **代理引擎** | 决策、调用模型、编排工具 | 模型提供商 + 系统提示 |
| **技能 (Skills)** | 代理能调用的能力 | 网页摘要、日报生成、代码审查 |
| **工作区 (Workspace)** | 代理的上下文和记忆 | AGENTS.md、结构化日志、文件 |
| **输出目标** | 结果落点 | 回发群聊、写入文档、触发 webhook |

## 普通聊天 vs OpenClaw 工作流：两种思维方式

下面用同一个场景对比两种用法的差异。

**场景：每天早上整理行业动态，发给团队**

### 普通聊天做法

```
你 → ChatGPT："帮我总结一下今天的 AI 新闻"
↓
ChatGPT 返回一段文字
↓
你复制粘贴发到群里
↓
明天重复
```

问题：完全依赖手动触发，结果格式不固定，无法扩展，明天还得做一遍。

### OpenClaw 工作流做法

```
定时任务（每天 8:00）
→ 触发 OpenClaw 代理
→ 调用"新闻摘要"技能（指定来源）
→ 按固定模板格式化输出
→ 自动发送到飞书群
→ 记录到工作区日志
```

结果：每天自动执行，格式一致，可追溯，不需要人工干预。

## 它适合解决什么问题

OpenClaw 最适合的是这类任务：

- 有固定触发入口，比如飞书、Telegram、定时任务、CLI。
- 有相对稳定的执行路径，比如"收集信息 -> 整理 -> 输出到群聊或文档"。
- 有明确结果，比如摘要、日报、告警、草稿、同步动作。
- 有持续复用价值，不是只做一次的临时演示。

### 典型入门场景

**场景 1：每日 AI 简报**

```
触发：定时（每天 9:00）
执行：抓取指定 RSS 源 → 提取关键信息 → 按模板生成简报
输出：发送到飞书群 #daily-briefing
```

**场景 2：会议纪要整理**

```
触发：用户发送会议录音或文字记录到 Telegram
执行：提取关键决策、行动项、负责人
输出：回发结构化纪要 + 可选写入飞书文档
```

**场景 3：链接摘要**

```
触发：用户在群里发一条链接
执行：抓取页面 → 识别语言 → 生成中文摘要
输出：直接回复在同一条消息下
```

**场景 4：定时巡检告警**

```
触发：每 30 分钟检测一次
执行：调用目标 API → 检查状态码和响应时间
输出：正常静默，异常时发飞书通知 + 记录日志
```

## OpenClaw 工作流的核心概念

在你开始配置之前，先熟悉这几个概念：

### 代理 (Agent)

代理是 OpenClaw 的执行主体。它接收渠道的输入，决定调用哪些技能，生成输出，并把结果发回对应位置。

每个代理都有：
- **系统提示 (system prompt)**：定义它的行为边界和角色
- **可用技能列表**：它被允许调用什么能力
- **工作区访问权限**：它能读取和写入哪些上下文

### 渠道 (Channel)

渠道是代理接收消息的入口。不同渠道有不同的配置方式：

- **飞书**：通过企业自建应用获取 App ID 和 App Secret，配置 webhook 回调地址
- **Telegram**：通过 BotFather 创建 bot，获取 bot token，配置 Webhook URL
- **CLI**：直接在终端调用，适合本地测试和开发

### 技能 (Skills)

技能是代理能调用的具体能力。OpenClaw 内置了一批技能，也支持自定义：

- `web-search`：网页搜索
- `summarize`：内容摘要
- `read-file`：读取工作区文件
- `write-file`：写入工作区文件
- 自定义技能：通过 API 调用外部服务

### 工作区 (Workspace)

工作区是代理的持久化上下文。最关键的文件是 `AGENTS.md`，它定义了代理的行为规则、权限边界和任务说明。

```markdown
# AGENTS.md 示例片段

## 角色定义
你是一个负责整理和分发信息的助手。

## 行为规则
- 收到链接时，自动生成中文摘要
- 不主动猜测用户意图，不明确时先确认
- 每次执行完毕，在工作区日志中记录时间和输出摘要

## 权限范围
- 可以读取 workspace/briefings/ 目录
- 可以写入 workspace/logs/ 目录
- 不可以发送通知到 #general 以外的频道
```

## 它不是靠"多聊几句"变强的

OpenClaw 的能力上限，往往不取决于你会不会写花哨提示词，而取决于下面几件事有没有接好：

- **入口有没有打通**：谁来触发它，消息从哪进来。
- **能力有没有装上**：它有没有你需要的技能、命令或外部集成。
- **规则有没有说明白**：它该大胆执行什么，该先确认什么。
- **输出有没有落点**：结果是回群里、写文档、发通知，还是存到工作区。
- **链路能不能复现**：这条流程下次还能不能稳定再跑。

这也是为什么很多人"装好了但觉得没啥用"。他们完成了安装，却还没有完成系统设计。

## 开始前先问自己这 4 个问题

如果下面四个问题你都能回答清楚，OpenClaw 通常就会比普通聊天工具更合适：

1. **这件事是不是会重复发生？** 如果只做一次，不值得配置。
2. **我是不是希望它通过某个渠道或命令被触发？** 还是每次都要你手动复制粘贴。
3. **我是不是需要它调用工具、技能或工作区文件？** 纯聊天不需要 OpenClaw。
4. **我是不是希望它产出一个可检查、可分发、可沉淀的结果？** 还是随便说说就行。

如果答案大多是"是"，那你现在做的就不是"玩一个 AI 工具"，而是在搭一条可以被反复调用的 AI 工作流。

## 新手最常见的误区

最常见的问题不是不会装，而是认知顺序错了：

| 错误顺序 | 正确顺序 |
|---------|---------|
| 先研究模型参数，后想清楚任务 | 先定义任务和预期输出 |
| 先装很多技能，后验证最小链路 | 先跑通一条最短闭环 |
| 先追求"什么都能做" | 先证明一件事能稳定跑 |
| 先做复杂多 Agent 系统 | 先让第一条消息完整闭环 |

这种顺序的问题在于，看起来进度很快，实际上没有任何一项被验证。

## 这一页读完后，你应该确认什么

继续往下之前，至少确认下面三件事：

- 你想做的是一个可重复的任务，不是一次性演示
- 你需要的是"输入 -> 执行 -> 输出"的稳定链路，不只是聊天
- 你的第一目标不是做大，而是先跑通一条最短闭环

## 更好的下一步

认知建立以后，下一步应该进入**最小安装**。目标不是一次性搭完整架构，而是先证明系统能跑通一条健康链路：进程能启动、模型能认证、日志能看到。
