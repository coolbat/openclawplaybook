第一次搭 OpenClaw 时，最容易发生的不是“撞上罕见 bug”，而是顺序错了、变量太多、验证不完整。于是一个本来很普通的问题，会被折腾成看起来像系统整体不稳定。

这篇文章的目的，就是帮你识别那些最常见、最浪费时间的 first-run 陷阱。

## 大部分 first-run 失败都很普通

这其实是好事。因为它说明问题往往是可拆解的，而不是系统坏掉了。

第一次失败最常见的来源通常只有几类：

- 环境还没装全
- 模型 key 或 provider 配错
- 渠道没真正接通
- 一次改太多导致证据被冲掉
- 工作流范围太大，把多个问题耦合在一起

也就是说，你现在最需要的不是更强的直觉，而是更稳的验证顺序。

## 陷阱一：健康检查还没通过，就开始叠加功能

这是最常见的一种。系统刚装完，模型还没验证，日志也没看懂，就开始：

- 接多个渠道
- 批量装技能
- 配定时任务
- 试浏览器自动化

结果是只要任何一层出错，你都不知道该从哪拆。

更好的做法是：先把“进程、模型、日志”这三件事跑通，再往上加东西。

## 陷阱二：一次改很多变量

看到一次失败后，很多人会同时：

- 换模型
- 换 provider
- 改 prompt
- 重装技能
- 重配渠道

这种做法最糟的地方，不是一定改错，而是你会失去唯一有价值的证据：**到底是哪一个变化真正影响了结果。**

first-run 阶段最重要的原则就是：一次只改一个变量。

## 陷阱三：机器人沉默，就先怀疑模型

机器人不回消息，可能是模型问题，但也可能是：

- 渠道事件没进来
- 账号权限不对
- token 配错
- 进程根本没在跑
- 结果回发失败

如果你一看到沉默就开始切模型，很可能是在错误层面排错。

## 陷阱四：把复杂工作流当作第一条验证路径

很多人第一条流程就做成：

- 多渠道输入
- 多技能串联
- 多个外部 API
- 复杂格式输出

一旦失败，根本无法定位到底是触发、模型、技能、权限还是输出链路出问题。

第一条工作流应该小到能被完整看清。复杂系统应该建立在最小成功样本之上。

## 陷阱五：把“Connected”当成真正成功

后台页面显示连接成功，不代表真实链路已经通了。

真正的成功标准应该是：

- 你发出一条真实测试消息
- 系统真的收到了
- 它真的执行了
- 它真的把结果回给你

只要这一步没发生，所谓连接成功都只能算“表面健康”。

## 一套更稳的检查顺序

如果你现在已经有点乱了，直接回到这个顺序：

1. 进程是否正常运行
2. provider 凭证是否有效
3. 日志是否有明确错误
4. 渠道是否能完成一次真实往返
5. 一个最小工作流是否能跑通

这个顺序的好处是，每一步都在缩小问题范围，而不是扩大猜测范围。

## 如何判断问题是“没配完整”还是“真的坏了”

可以用一个很实用的判断法：

如果你现在还说不清问题属于下面哪一层，那它大概率还只是“没配完整”：

- 安装层
- 模型层
- 渠道层
- 工作流层

真正难的 bug，通常是在你已经明确知道层级之后才会出现。大多数新手第一阶段遇到的，都还没到那个级别。

## 新手最该养成的习惯

每次失败时，至少记住三件事：

- 你刚刚做了什么
- 你看到了什么现象
- 你改过什么变量

很多人以为自己是在排错，实际上是在不断清空现场。

## 一条更好的 first-run 心态

不要把第一次搭建当成“必须一次成功的正式上线”。把它当成一个建立认知的过程：

- 先证明系统活着
- 再证明一层层链路是真的通
- 最后再追求效率、漂亮和自动化程度

只要顺序对了，你走得反而更快。
