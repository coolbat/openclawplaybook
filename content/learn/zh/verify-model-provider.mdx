新手搭环境时，一个非常高频的错误是：安装、渠道、技能、模型、工作流一起推进。最后一旦失败，你根本分不清问题出在哪一层。

更稳的做法是先把模型服务单独验证掉。因为模型是整条链路里最基础的依赖之一，如果这一层没通，后面的渠道和工作流都只是在更高层放大问题。

## 为什么只验证一个 provider

原因很简单：你现在需要的是确定性，不是选择自由。

如果你一开始同时接：

- 两个 provider
- 几把不同 key
- 不同接口类型
- 不同模型别名

那么一旦报错，你很难判断到底是：

- key 错了
- provider 配错了
- 模型名称写错了
- 接口类型不匹配
- fallback 在偷偷接管

所以 first-run 阶段的最佳策略就是：**一个 provider、一把 key、一条最短响应链路。**

## 什么叫“验证成功”

不是配置文件里写上了 provider 名称，也不是后台页面看起来像是连上了。

真正的验证成功，至少要同时满足三件事：

- 请求真的发出去了
- provider 真的返回了结果
- 你知道当前返回结果是哪个模型给的

最后这一点非常重要，因为很多人以为自己在用 A，实际却因为 fallback 或默认配置在用 B。

## 推荐的验证顺序

最稳的顺序通常是：

1. 只配置一个 provider
2. 只填一把 key
3. 只选一个你确定支持的模型
4. 跑一个最简单的测试请求
5. 看日志确认实际命中的 provider 和模型

在这一层，越简单越好。不要一上来就用复杂工作流测试模型。模型验证应该是系统里最短的一段链路。

## 为什么“看日志”比“看到回复”更重要

只看回复会遗漏两个高频问题：

- 你以为走的是某个模型，实际走的是 fallback
- 你以为响应正常，实际只是默认路径兜底了

而日志能告诉你更关键的事实：

- 真正命中了哪个 provider
- 真正使用了哪个模型
- 报错发生在认证、网络还是响应阶段

所以“有回复”只是第一层，“知道这条回复到底从哪里来的”才是真正完成验证。

## 一条最短测试链路应该长什么样

你当前要的不是效果最好，而是最容易定位问题。

一个合格的测试链路通常满足：

- 不依赖渠道
- 不依赖技能
- 不依赖长上下文
- 不依赖复杂输出格式

换句话说，它只验证“模型服务这一层到底活不活”。

## 新手最常见的误区

### 误区一：一开始就配置多个 provider

看起来像给自己留后路，实际上是在给排错制造分叉。

### 误区二：看到返回结果就结束验证

这会忽略模型错配、fallback 接管和隐性路由问题。

### 误区三：模型层还没稳定，就开始调 prompt

这会把“链路问题”误判成“回答问题”。

### 误区四：把复杂工作流当模型验证工具

如果测试链路本身依赖很多外部系统，你根本不知道失败点到底在哪。

## 一个很实用的判断句

如果你现在不能清楚回答下面这句话，就还不算完成模型验证：

> 我知道当前是哪家 provider、哪个模型、通过哪条路径返回了这次结果。

只要这个答案还是模糊的，后面的工作流成功也会带着不确定性。

## 什么时候才适合加第二个 provider

等你满足下面两个条件以后，再去加第二个 provider 才划算：

- 第一个 provider 已经稳定工作
- 你已经知道如何从日志里确认真正命中的模型

这样第二个 provider 带来的是冗余和灵活性，而不是新的混乱。
